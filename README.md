# Notifier-Service

## Архитектурные решения

1. **Worker pool**
   - Используется фиксированное количество воркеров, задаётся при создании `Notifier`
   - Воркеры запускаются в `NewNotifier()` и работают до вызова `Close()`

2. **Rate limiting**
   - Rate limiter реализован с помощью `golang.org/x/time/rate`
   - При превышении лимита воркеры ждут с помощью `limiter.Wait()`

3. **Retry**
   - При получении 429 (Too Many Requests) от внешнего сервиса запрос повторяется (максимум 3 раза)
   - Backoff вычисляется как `baseDelay * 2^attempt + rand jitter`, с jitter 0.1 (Максимум: 1 минута+jitter)

4. **Graceful shutdown**
   - `Close()` дожидается завершения всех текущих задач с помощью `sync.WaitGroup`, `wg.Wait()`
   - Используется атомарный флаг `qClosed`, при нём, последующие вызовы `Close()` возвращают ошибку
   - Метод `Send()` перестаёт принимать новые сообщения

5. **Метрики**
   - Реализована структура `Stats` с полями `Sent`, `Failed`, `Retries`
   - Все счётчики атомарные (`atomic.Int64`) для безопасной работы в многопоточной среде

6. **Abort operation**
   - При истёкшем/отменённом контексте отменить работу:\
      `
            select {
         case <-ctx.Done():
            return ctx.Err()
         default:
         }
      `

## Ограничения

- При ошибках, отличных от 429, ретрая нет
- Не сохраняются логи
- Нет таймаута на вызов `ExternalClient.Post()`
- Нет сброса/взятия метрик за определённый период
